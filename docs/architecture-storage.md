## 3. Data Model

### 3.1 SQLite Schema

```sql
-- Core memory table
CREATE TABLE memories (
  id          TEXT PRIMARY KEY,           -- mem_<ulid>
  content     TEXT NOT NULL,              -- raw memory text
  source      TEXT NOT NULL DEFAULT 'user', -- 'user' | 'agent' | 'system' | 'host_memory_write'
  session_id  TEXT,                       -- session this memory was created in
  project     TEXT,                       -- optional project namespace
  tags        TEXT DEFAULT '[]',          -- JSON array of tags
  metadata    TEXT DEFAULT '{}',          -- JSON: {importance, scope, valid_until, ...}
  created_at  INTEGER NOT NULL,           -- unix epoch ms
  updated_at  INTEGER NOT NULL            -- unix epoch ms
);

-- FTS5 index (keyword search)
CREATE VIRTUAL TABLE memories_fts USING fts5(
  content, tags,
  content='memories', content_rowid='rowid'
);

-- Triggers to keep FTS5 in sync
CREATE TRIGGER memories_ai AFTER INSERT ON memories BEGIN
  INSERT INTO memories_fts(rowid, content, tags) VALUES (new.rowid, new.content, new.tags);
END;
CREATE TRIGGER memories_ad AFTER DELETE ON memories BEGIN
  INSERT INTO memories_fts(memories_fts, rowid, content, tags) VALUES ('delete', old.rowid, old.content, old.tags);
END;
CREATE TRIGGER memories_au AFTER UPDATE ON memories BEGIN
  INSERT INTO memories_fts(memories_fts, rowid, content, tags) VALUES ('delete', old.rowid, old.content, old.tags);
  INSERT INTO memories_fts(rowid, content, tags) VALUES (new.rowid, new.content, new.tags);
END;

-- Vector index (default: text-embedding-v3, 1024d; local ONNX optional: 768d)
CREATE VIRTUAL TABLE memories_vec USING vec0(
  id TEXT PRIMARY KEY,
  embedding FLOAT[1024]
);

-- Enrichment records (L1 insights)
CREATE TABLE insights (
  id          TEXT PRIMARY KEY,
  memory_id   TEXT NOT NULL REFERENCES memories(id),
  content     TEXT NOT NULL,
  layer       TEXT NOT NULL DEFAULT 'L1', -- 'L1' | 'L2' | 'L3'
  metadata    TEXT DEFAULT '{}',
  created_at  INTEGER NOT NULL,
  UNIQUE(memory_id, layer)
);

-- Preference records (compounding)
CREATE TABLE preferences (
  id          TEXT PRIMARY KEY,
  body        TEXT NOT NULL,
  tier        TEXT NOT NULL DEFAULT 'trial', -- 'trial' | 'confirmed' | 'retired'
  confidence  REAL NOT NULL DEFAULT 0.0,
  evidence    TEXT DEFAULT '[]',
  origin      TEXT,
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL
);

-- Preferences evidence log
CREATE TABLE preference_evidence (
  id          TEXT PRIMARY KEY,
  pref_id     TEXT NOT NULL REFERENCES preferences(id),
  memory_id   TEXT NOT NULL REFERENCES memories(id),
  result      TEXT NOT NULL,              -- 'applied' | 'violated' | 'outdated'
  session_id  TEXT,
  created_at  INTEGER NOT NULL
);

-- Persona snapshots
CREATE TABLE personas (
  id          TEXT PRIMARY KEY,
  profile     TEXT NOT NULL,
  content     TEXT NOT NULL,
  version     INTEGER NOT NULL DEFAULT 1,
  created_at  INTEGER NOT NULL
);

-- Sessions
CREATE TABLE sessions (
  id          TEXT PRIMARY KEY,
  agent       TEXT,
  project     TEXT,
  started_at  INTEGER NOT NULL,
  ended_at    INTEGER,
  metadata    TEXT DEFAULT '{}'
);

-- Indexes
CREATE INDEX idx_memories_created ON memories(created_at);
CREATE INDEX idx_memories_tags ON memories(tags);
CREATE INDEX idx_memories_session ON memories(session_id);
CREATE INDEX idx_insights_memory ON insights(memory_id);
CREATE INDEX idx_preferences_confidence ON preferences(confidence);
CREATE INDEX idx_pref_evidence_pref ON preference_evidence(pref_id);
```

### 3.4 Two-Way Sync: Vault ↔ SQLite

**Token overhead: 0.** Sync = file I/O + mtime comparison. Ga ada LLM call.

| File | Human edit? | Direction | Overwrite behavior |
|---|---|---|---|
| `persona.md` | ✅ Ya | two-way | Bridge baca dari SQLite, merge human edit |
| `preferences/pref-*.md` | ✅ Ya | two-way | Dream baca dari sini, update confidence |
| `active.md` | ❌ Jangan | one-way (SQLite→vault) | Dream regenerate dari SQLite tiap jam |
| `inbox/sig-*.md` | ❌ Jangan | one-way (SQLite→vault) | Write-through dari `remember` |
| `log/*.md` | ❌ Jangan | append-only | Immutable |

**Race condition resolution:**

```
Problem: Human edit + Bridge write at same second → edit lost.

Solution: Last-write-wins with fallback:
  1. Before Bridge write: check vault mtime vs SQLite
  2. If vault.mtime > SQLite.updated_at → human edit detected
     → sync vault → SQLite FIRST
     → then run Bridge (Bridge reads from SQLite, not vault)
  3. If SQLite.updated_at > vault.mtime → Bridge already wrote
     → merge: Bridge content + human edits (diff-based)
  4. If both changed → conflict marker written to vault
     → human resolves via `capricorn resolve`
```

**active.md role in v2:**
- **Internal state** — Dream reference untuk bandingin preferences baru vs existing
- **Human audit** — satu file baca semua confirmed preferences
- **Recovery** — kalau SQLite corrupt, active.md bisa jadi restore point
- **NOT injected** — context injection via `capricorn.context` tool, bukan passive file read

## 4. Storage Engine (Sync)

### 4.1 Write Path: `capricorn remember`

```
Agent: capricorn remember "User prefers dark mode" --tags preference,ui --importance 0.9

  1. VALIDATE
     └─ coerce args, generate ULID, set timestamp

  2. SQLite WRITE
     └─ INSERT INTO memories (id, content, tags, metadata, ...)

  3. FTS5 INDEX (auto via trigger)
     └─ memories_fts updated

  4. VECTOR INDEX (runs at `remember` time)
     └─ L0 embedding is computed synchronously when `remember` is called
     └─ After embedding: vector table populated, hybrid recall enabled

  5. VAULT WRITE-THROUGH
     └─ write Brain/inbox/sig-{timestamp}-{slug}.md

  6. RESPONSE
     └─ { id: "mem_7f3a...", status: "stored" }
```

### 4.2 Read Path: `capricorn recall`

```
Agent: capricorn recall "user interface preferences" --top-k 5

  1. PARALLEL SEARCH
     ├─ FTS5:    SELECT rowid, rank FROM memories_fts WHERE content MATCH ?
     ├─ Vector:  SELECT id, distance FROM memories_vec WHERE embedding MATCH ? LIMIT 20
     └─ Full:    SELECT id, content, tags, created_at FROM memories WHERE ...

  2. RRF FUSION (k=60)
     └─ score = Σ 1/(k + rank_i) across FTS5 rank + vector rank

  3. DEDUP & RANK → top-k

  4. ENRICH (optional)
     └─ join with insights, preferences (if include_enriched=true)

  5. RESPONSE
     └─ [{ id, content, tags, score, created_at, insight?, preference? }, ...]
```

### 4.3 Hybrid Scoring Detail

```
RRF(rank_fts5, rank_vector, k=60):
  result_1: FTS5 rank=1, Vector rank=3 → 0.0164 + 0.0159 = 0.0322
  result_2: FTS5 rank=5, Vector rank=1 → 0.0154 + 0.0164 = 0.0318
  result_3: FTS5 rank=2, Vector rank=2 → 0.0161 + 0.0161 = 0.0323 ← WINNER
  RRF favors agreement: ranked high in BOTH systems wins.
```

### 4.4 Vector Index Options

| Option | Approach | Pros | Cons |
|---|---|---|---|
| **API (text-embedding-v3)** | Cloud LLM, 1024d | Highest accuracy, zero local cost, instant | Needs API key, network |
| **sqlite-vec** | SQLite extension, brute-force | Zero deps, fast for <100K | O(n) scan, no ANN |
| **HNSW (usearch)** | External lib, ANN index | ~45ms at 100K, scales to millions | Extra dep, memory |
| **ONNX (local)** | Embedded model, 768d | Offline, privacy | 188MB model, slower cold start |

**v2 default:** API (text-embedding-v3, 1024d). Local ONNX (768d) optional fallback.

**Two embedding models, different purposes:**

| Model | Dims | Purpose | Token |
|---|---|---|---|
| **text-embedding-v3** (API) | 1024d | Storage: semantic search via `memories_vec` | API cost |
| **all-MiniLM-L6-v2** (local) | 384d | Validation: HyperTune + HaluGard scoring | 0 |
| **EmbeddingGemma Q4** (ONNX) | 768d | Optional: offline storage replacement | 0 |

### 4.5 Offline Mode

```
DEFAULT (API): embed via text-embedding-v3 → 1024d
OPTIONAL (Local): capricorn enable-vector --local → ONNX 768d, fully offline
FALLBACK: no API key, no local model → FTS5-only recall
```

**Async enrichment gap — not a problem:**
- Bridge (6h): Forge L1→L3 enrichment
- Dream (1h): Compound preferences + regenerate active.md
- Memory tetap accessible via FTS5 recall setiap saat
- Enrichment = bonus intelligence, bukan core function
- Gap 6h wajar karena Forge pake API (cost control)
